## 技术栈总览（小白友好版）

> **答辩必背**：老师问"你用了哪些技术"，直接回答：
> "后端：Spring Boot + MyBatis-Plus + MySQL + Redis + JWT + WebSocket/STOMP；

#### 1. Spring Boot 2.7.9 —— 项目的"骨架"

**一句话理解**：就像盖房子需要钢筋混凝土框架，Spring Boot 就是整个后端程序的骨架，它帮你把各个零件自动组装起来，你不用操心谁先启动、谁依赖谁。

**解决了什么问题**：传统 Java Web 开发要手写大量 XML 配置文件来"组装"各个组件，Spring Boot 的"自动配置"机制会根据你引入的依赖自动完成 90% 的配置工作。比如你引入了 MySQL 驱动，它自动帮你配好数据库连接池，无需手写一行配置。

**在本项目中的具体作用**：
- **内嵌 Tomcat 服务器**：传统做法要单独下载 Tomcat、部署 war 包；Spring Boot 把 Tomcat 打包进 jar 里，一个 `java -jar` 命令就能启动，就像把发动机直接装进了车身
- **注解驱动开发**：写上 `@RestController` 就表示这是一个 API 接口类；写上 `@Service` 就表示这是业务逻辑类；Spring Boot 自动扫描这些注解并管理它们的生命周期
- **声明式事务 `@Transactional`**：在下单方法上加上这个注解，如果扣库存成功了但插订单失败了，之前扣掉的库存会自动"回滚"恢复，保证数据一致性
- **健康检查**：`/actuator/health` 接口可以检查数据库连不连得上、Redis 活没活着，运维人员用这个接口就能判断系统是否健康

#### 2. MyBatis-Plus 3.5.3 —— Java 对象和数据库之间的"翻译官"

**一句话理解**：数据库里存的是一行行的表数据，Java 程序里用的是一个个对象。MyBatis-Plus 就像一个翻译官，自动帮你把"数据库的行"翻译成"Java 对象"，反之亦然。

**解决了什么问题**：没有它的话，你需要手写大量的 SQL 语句来"把用户输入的年龄存到 user 表的 age 字段"，每个增删改查都要写几十行重复代码。MyBatis-Plus 的 `BaseMapper` 自带 `selectById()` / `insert()` / `updateById()` / `deleteById()` 四个基本操作，一行代码都不用写。

**在本项目中的具体作用**：
- **通用 CRUD**：`userMapper.selectById(1L)` 一行代码等价于 `SELECT * FROM user WHERE id = 1`，不用写 SQL
- **分页插件**：`new Page<>(1, 20)` 传入 Mapper 方法，自动拼接 `LIMIT 0, 20` 并返回总条数、总页数
- **乐观锁插件**：下单扣库存时，用 `WHERE stock = #{旧库存值}` 的方式防止超卖（两条订单同时看到库存=10，都觉得自己能买，但实际只能卖一份）。我把版本号字段配置为乐观锁字段，更新时自动 `SET version = version + 1 WHERE version = #{旧版本号}`
- **复杂 SQL 用 XML**：简单查询靠 BaseMapper 自动生成，复杂的（比如"查某个用户报名了哪些活动"要联 activity 和 activity_participant 两张表）就手写在 Mapper XML 里，灵活可控

#### 3. MySQL 8.0 —— 数据的"仓库"

**一句话理解**：就像一个巨大的 Excel 表格，但能存几百万行数据，支持多人同时读写不冲突。

**解决了什么问题**：用户的账号密码、商品信息、订单记录、聊天消息这些数据都存在 MySQL 里。它比 Excel 强的地方是：支持"事务"（一个操作包含多个步骤，要么全成功、要么全撤销），支持"索引"（给常用查询字段建索引，查 100 万条中的一条只需几毫秒），支持"并发控制"（100 个人同时下单不会互相干扰）。

**在本项目中的具体作用**：
- **37 张业务表**：user（用户）、product（商品）、orders（订单）、cart（购物车）、chat_message（聊天消息）等，覆盖全业务域
- **InnoDB 引擎**：支持行级锁（只锁被操作的那一行，不锁整张表）和事务（下单时扣库存+插订单+清购物车三个操作在一个事务里，要么全成功要么全失败）
- **REPEATABLE READ 隔离级别**：MySQL 默认级别，保证同一事务内多次读同一行数据返回的值一致，防止"我明明看到还有 5 件库存，点购买时却变成 0 了"的幻读现象
- **FULLTEXT 全文索引**：给商品名称建了全文索引，搜索"有机苹果"时比 `LIKE '%有机苹果%'` 快几十倍

#### 4. Redis 5.x —— 数据的"高速缓存"

**一句话理解**：MySQL 是硬盘上的仓库（容量大但取货慢），Redis 是收银台旁的抽屉（容量小但伸手就能拿到）。把常用数据放 Redis 里，就不需要每次都去仓库翻找了。

**解决了什么问题**：如果每次有用户打开商品详情都去 MySQL 查一遍，1000 个人同时看同一个爆款商品，MySQL 要执行 1000 次相同的查询。把结果缓存到 Redis 里，设置 30 秒过期，30 秒内的 1000 次请求直接走 Redis 返回，MySQL 只查了一次。

**在本项目中的具体作用**：
- **商品详情缓存**：热门商品查一次后缓存 30 秒，后续请求直接走 Redis，响应速度从 100ms 降到 5ms
- **分类树缓存**：商品分类树（"生鲜 > 水果 > 苹果"）是一棵树形结构，构建一次要查多次数据库。启动时全量加载到 Redis，修改分类时主动清缓存
- **JWT 黑名单**：JWT 是无状态的（签发了就不能撤回），但如果用户点了"退出登录"，我们要让这个 token 立即失效。做法是把 token 的 ID 存到 Redis 里，设一个过期时间（等于 token 剩余有效期）。每次请求先查 Redis 黑名单，命中了就拒接
- **聊天在线状态**：用 Redis 的 Set 数据结构记录哪些用户在线。用户连上 WebSocket 时 `SADD online_users {userId}`，断开时 `SREM`。判断对方是否在线只需 `SISMEMBER`，是 O(1) 时间复杂度的操作
- **短信验证码**：验证码是临时数据，存 MySQL 浪费。存在 Redis 里，设 5 分钟 TTL（过期自动删除），校验时先查 Redis，成功就删除
- **接口限流计数器**：用 `INCR` 命令给每个 IP+ 接口的组合计数，第一次调用设 `EXPIRE 60`（60 秒后自动清零）。如果 1 分钟内计数超过限制（比如短信发送 3 次），直接返回"请求过于频繁"
- **未读消息计数**：用 `HINCRBY` 给每个会话的未读消息数 +1，用户进入会话时清零

#### 5. Spring Security —— 系统的"门禁"

**一句话理解**：就像写字楼门口的保安——进门前查你有没有工牌（认证），进门后查你有没有权限去这个楼层（授权）。

**解决了什么问题**：没有安全框架的话，每个接口都要手动写"检查用户有没有登录、检查用户有没有权限"的代码，而且容易漏配。Spring Security 用配置的方式统一处理，一个接口要么声明为"公开"，要么就强制要求登录，不会漏网。

**在本项目中的具体作用**：
- **无状态 JWT 会话**：传统做法是登录后服务器存 session，这要求每次请求都带着 sessionId，服务器要查内存里的 session。我设置了 `SessionCreationPolicy.STATELESS`，告诉 Spring Security"别创建 session，所有身份信息都在 JWT token 里"，这样服务器不存任何状态，方便以后做分布式部署
- **BCrypt 密码加密**：用户注册时输入的密码不是明文存储的，而是经过 BCrypt 算法加盐哈希后的密文（`$2a$10$...`）。登录时 `matches(输入的密码, 存储的密文)` 比较，即使数据库泄露，黑客也拿不到原始密码
- **跨域 CORS 配置**：前端跑在 localhost:5173，后端跑在 localhost:8081，浏览器的安全策略默认禁止跨端口访问。我在 `CorsConfigurationSource` 里把 5173、5174、5175、5176、3001 这几个前端端口全放行了
- **公开接口白名单**：`/api/auth/**`（登录注册相关）、`/api/captcha/**`（验证码）、`/api/sms/**`（短信）、`/api/product/list` 等浏览商品的接口标记为 `permitAll()`，不登录也能访问。其余接口 `authenticated()` 全部要求登录

#### 6. JWT（JSON Web Token）—— 用户的"电子身份证"

**一句话理解**：就像身份证——上面印了你的姓名、身份证号、签发日期，防伪水印确保不可伪造。JWT 就是一段加密的字符串，里面装着用户 ID、用户名、角色，服务器看到它就知道"你是谁"。

**解决了什么问题**：传统做法用 Session：用户登录后服务器创建一条 session 记录，返回一个 sessionId。下次请求带着 sessionId 来，服务器查内存找到对应的 session。问题是服务器重启 session 全丢；两台服务器要共享 session；移动端 App 不方便用 cookie。JWT 把信息加密后直接放在 token 里，服务器不存任何东西，解析即可，天生适合前后端分离和分布式。

**在本项目中的具体作用**：
- **生成 token**：用户登录成功后，用 `Jwts.builder()` 把 userId、username、userType 三个字段打包到 payload 里，用 HS256 算法 + 密钥签名，设 7 天有效期。生成的 token 是一串像 `eyJhbGciOiJIUzI1NiJ9.eyJ1c2VySWQiOjEwMDAxfQ.xxx` 的字符串
- **验证 token**：每次请求从 `Authorization: Bearer xxx` 头里取出 token，用同一个密钥解析，如果是伪造的或已过期的，解析失败返回 401
- **Payload 里放了 userType**：0=买家、1=商家、2=管理员。权限拦截器解析 token 后拿到 userType，就能判断这个人有没有资格访问这个接口

#### 7. WebSocket + STOMP —— 实时聊天的"对讲机"

**一句话理解**：普通的 HTTP 请求就像寄信——客户端写信（发请求），服务器回信（返回响应），一次就结束了。WebSocket 就像对讲机——建立连接后双方随时可以说话，不需要等对方先开口。

**解决了什么问题**：聊天场景下，HTTP 轮询是"每隔 2 秒去服务器问'有我的新消息吗？'"，99% 的请求都是空返回，浪费资源。WebSocket 是"服务器有消息了主动推给你"，客户端不用反复轮询。

**在本项目中的具体作用**：
- **STOMP 子协议**：原生 WebSocket 只是"建立了连接"，没有消息格式和路由概念。STOMP 给它加上了"订阅/发布"模型——客户端订阅 `/topic/chat/123` 这个频道，服务器向这个频道发消息，所有订阅者都能收到。就像一个微信群，发到群里所有人都能看到
- **点对点通知**：`/queue/notifications` 是用户专属频道，服务器向这个频道发消息，只有这个用户自己能收到。用于推送"您的订单已发货"、"客服已接入"等通知
- **SockJS 降级**：WebSocket 依赖 HTTP Upgrade 机制，某些老旧的网络代理会拦截这个升级请求。SockJS 会自动检测——WebSocket 能用就优先用，用不了就降级到"长轮询"（发一个请求，服务器不立即返回，等有消息了再返回），对用户完全透明
- **握手鉴权**：STOMP 连接建立时也有类似 HTTP 的"握手"阶段，我在 `ChannelInterceptor` 里拦截 CONNECT 帧，验证 JWT token 的合法性，防止匿名用户建 WebSocket 连接偷听聊天内容
- **消息先落库再推送**：收到消息后先 `INSERT` 到 `chat_message` 表，成功后再推送给对端。这样即使推送失败（比如对端正好断网），消息已经存在数据库里了，下次上线拉取离线消息时能补回来

#### 8. SseEmitter —— AI 回答的"传真机"

**一句话理解**：WebSocket 是双向对讲机，SSE（Server-Sent Events）是单向传真机——服务器单方面向客户端推送数据，客户端不需要回话。适合 AI 回答这种"你问一句，我一句一句往外蹦答案"的场景。

**解决了什么问题**：如果等 DeepSeek 完全生成完 500 字的回答再一次性返回给用户，用户要干等 5 秒看到一片空白，体验很差。用 SseEmitter 后，DeepSeek 那边生成一个字，服务器就推一个字给前端，用户看到的是 typing... 的打字机效果，感知延迟大幅降低。

**在本项目中的具体作用**：
- **`MediaType.TEXT_EVENT_STREAM`**：告诉浏览器"我要用 SSE 协议推送数据"，前端 `new EventSource(url)` 就能接收
- **异步线程池**：`@Async` 注解把调用 DeepSeek 的工作扔到一个独立的线程池里执行，Tomcat 的主线程立即返回 emitter 对象，不会傻等着 DeepSeek 慢慢生成。如果不用异步，Tomcat 默认只有 200 个线程，100 个人同时问 AI，100 个线程全被卡住，其他人连商品列表都打不开了
- **30 秒超时**：万一 DeepSeek 那边卡住了，SseEmitter 30 秒后自动断开，前端 EventSource 收到 error 事件可以提示用户重试
- **前端打字机效果**：`es.addEventListener('message', e => { answerText += e.data })`，每次收到一个文字片段就追加到页面上

#### 9. DeepSeek API —— AI 客服的"大脑"

**一句话理解**：DeepSeek 是一个"大语言模型"，就像一个读了整个互联网知识的人。你问它问题，它能根据它学过的知识给你回答。

**解决了什么问题**：传统客服要么是人工（成本高、响应慢），要么是关键词匹配机器人（"输入退货，回复'请拨打 400 电话'"），体验很差。DeepSeek 能理解自然语言，你问"这个苹果甜不甜"和问"这个水果口感怎么样"它能理解是同一个意思。

**在本项目中的具体作用**：
- **System Prompt 约束范围**：在每次请求前先"告知"DeepSeek 它的角色是"'乡农优选'商城的 AI 客服，只回答商品咨询、订单状态、活动优惠、售后政策、平台规则这 5 类问题，其他问题礼貌拒绝"。这样老师问"今天天气怎么样"不会被带偏
- **流式返回**：DeepSeek 支持 `stream: true` 参数，不一次性返回完整答案，而是逐 token（词块）推送。配合 SseEmitter 实现流畅的打字机体验
- **HttpClient 连接池**：用 Apache HttpClient 维护一个与 DeepSeek API 的连接池，复用 TCP 连接，避免每次 AI 对话都三次握手建立新连接的开销

#### 11. Lombok —— 代码的"省力助手"

**一句话理解**：Java 里每个字段都要写 `getXxx()` 和 `setXxx()` 方法，一个 20 个字段的 User 类就要写 40 个方法 + 构造函数 + `toString()` = 几百行。Lombok 的 `@Data` 注解一行搞定——编译时自动生成所有 getter/setter/toString/equals/hashCode。

**在本项目中的具体作用**：
- Model 层 37 个实体类全部用 `@Data`，省去数千行样板代码
- `@Slf4j` 注解替代 `private static final Logger log = LoggerFactory.getLogger(...)`，一行变一行但省掉类名拼接
- 代码更干净，阅读时不会被机械重复的 getter/setter 干扰

#### 12. SLF4J + Logback —— 系统的"黑匣子"

**一句话理解**：飞机上有黑匣子记录所有飞行数据，出了事故通过黑匣子还原过程。SLF4J + Logback 就是程序的"黑匣子"——记录每个请求的来龙去脉，出问题时通过日志快速定位。

**在本项目中的具体作用**：
- **统一日志格式**：`2026-06-05 23:01:42.714 ERROR [main] [traceId=abc123] ... — 连接丢失`，包含时间戳、线程名、traceId、日志内容
- **traceId 全链路追踪**：`TraceIdFilter` 在请求进来时生成一个唯一 traceId，注入到日志的 MDC 中。同一个请求经过 Controller → Service → Mapper 的所有日志都带相同的 traceId，用 `grep traceId=abc123 logs/` 就能还原这个请求的完整轨迹
- **文件滚动策略**：日志文件超过 50MB 自动切新文件，保留最近 30 天，旧文件自动删除。防止日志把硬盘撑爆
- **分级控制**：生产环境设 INFO 级别，只记录重要事件；排查问题时临时改成 DEBUG 级别，可以看到详细的 SQL 语句和参数值

#### 14. JUnit 5 + Mockito —— 代码的"体检医生"

**一句话理解**：写完代码你怎么知道它对不对？JUnit 就是一套自动化"体检"流程——你写一段测试代码模拟用户行为（比如"下单"），然后断言"下单后库存减 1、订单状态为待支付、购物车已清空"，如果实际结果和预期不一致，测试就报红。

**在本项目中的具体作用**：

- `@SpringBootTest`：启动完整的 Spring 容器，模拟真实环境测试
- `@WebMvcTest(AdminChatController.class)`：只加载 Controller 层，Mock 掉 Service 依赖，轻量快速
- `MockMvc`：用 Java 代码模拟 HTTP 请求，发一个 GET /api/product/list，断言返回的 JSON 里 `code == 0` 且 `data.list.size() > 0`
- `@MockBean`：把真实的 Service 换成"替身"，比如测试 Controller 时不需要真的连数据库，Mock 一个 Service 返回假数据就行

#### 15. Jackson —— JSON 的"翻译器"

**一句话理解**：Java 程序用的是对象（`User { id: 1, name: "张三" }`），前端需要的是 JSON（`{"id":1,"name":"张三"}`）。Jackson 就是两者之间的翻译器——自动把 Java 对象转成 JSON 字符串发给前端，把前端传来的 JSON 字符串转成 Java 对象。

**在本项目中的具体作用**：
- Spring Boot 内置 Jackson，`@RestController` 的方法返回一个对象，自动转成 JSON
- `@JsonFormat(pattern = "yyyy-MM-dd HH:mm:ss")`：统一日期格式，避免时间戳数字让人看不懂
- `ObjectMapper.readValue(jsonString, User.class)`：把前端 POST 的 JSON 请求体解析成 Java 对象

#### 16. Vue 3 (Composition API) —— 前端的"积木系统"

### 2.1 PermissionInterceptor（鉴权拦截器）

文件：`backend/src/main/java/com/example/minimall/interceptor/PermissionInterceptor.java`

这是答辩时被问到"你怎么做的权限控制"我重点展示的代码。
初版用"默认放行+白名单"，被测出漏洞后改成"默认拒绝+显式白名单"。

```java
// 公开接口白名单（不登录就能访问）
private static final Set<String> PUBLIC_PATHS = new HashSet<>(Arrays.asList(
    "/api/auth/login", "/api/auth/register",
    "/api/product/list", "/api/product/search",
    "/api/product/{id}",  // 商品详情
    "/api/category/list", "/api/category/{id}",
    "/api/ai/query"  // AI 客服允许匿名问
));

// 角色判断
private boolean isSellerAllowed(String uri) {
    return uri.startsWith("/api/seller/") 
        || uri.startsWith("/api/product/")  // 商家能管自己商品
        || uri.startsWith("/api/ai/");
}

private boolean isBuyerAllowed(String uri) {
    return uri.startsWith("/api/cart/")
        || uri.startsWith("/api/order/create")
        || uri.startsWith("/api/order/list")
        || uri.startsWith("/api/refund/");
}
```

```java
@Override
public boolean preHandle(HttpServletRequest req, 
                          HttpServletResponse resp,
                          Object handler) throws Exception {
    String uri = req.getRequestURI();
    String method = req.getMethod();

    // 1. OPTIONS 直接放行（跨域预检）
    if ("OPTIONS".equalsIgnoreCase(method)) {
        return true;
    }

    // 2. 公开接口放行
    for (String p : PUBLIC_PATHS) {
        if (matchPath(p, uri)) {
            return true;
        }
    }

    // 3. 拿 token
    String auth = req.getHeader("Authorization");
    if (auth == null || !auth.startsWith("Bearer ")) {
        resp.setStatus(401);
        resp.getWriter().write("{\"code\":1,\"message\":\"未登录\"}");
        return false;
    }
    String token = auth.substring(7);

    // 4. 解析 JWT
    Claims claims;
    try {
        claims = JwtUtil.parseToken(token);
    } catch (Exception e) {
        resp.setStatus(401);
        resp.getWriter().write("{\"code\":1,\"message\":\"token 无效\"}");
        return false;
    }
    Integer userType = claims.get("userType", Integer.class);
    Long userId = claims.get("userId", Long.class);
    req.setAttribute("userId", userId);
    req.setAttribute("userType", userType);

    // 5. 角色校验（默认拒绝）
    if (isAdminAllowed(uri)) {
        if (userType != 2) {  // 管理员
            resp.setStatus(403);
            resp.getWriter().write("{\"code\":1,\"message\":\"无权限\"}");
            return false;
        }
    } else if (isSellerAllowed(uri)) {
        if (userType != 1 && userType != 2) {  // 商家或管理员
            resp.setStatus(403);
            return false;
        }
    } else if (isBuyerAllowed(uri)) {
        if (userType == null) {
            resp.setStatus(403);
            return false;
        }
    } else {
        // 既不是公开、也不是任何角色的接口 → 默认拒绝
        resp.setStatus(403);
        resp.getWriter().write("{\"code\":1,\"message\":\"接口不存在或无权访问\"}");
        return false;
    }

    return true;
}
```

**这段代码答辩时怎么讲：**
> "我之前用的是默认放行+白名单模式，结果安全测试 4 个用例都失败——无 token、
> 伪造 token、买家越权访问管理接口，全部 200 通过。问题在于白名单是配置的，
> 一旦漏配新接口就放行了。改完之后是默认拒绝+显式白名单：不在白名单的接口，
> 一定要校验 role 才能过，这样以后加新接口忘了配置也只会是 403，不会是 200。"

> "另外用 JWT 解析出 userType，把 userId 放 request attribute 里，
> Controller 直接 @RequestAttribute Long userId 就能拿到，比传统从 session
> 取参数干净。"

技术细节展开：
- `matchPath` 里 `{id}` 用了 `Pattern` 编译的正则匹配，
  避免 `uri.startsWith("/api/product/")` 把 `/api/productxxx` 也匹配上
- 401 和 403 区分：无 token/伪造 token → 401（身份问题），有 token 但 role 不对 → 403（权限问题）
- preHandle 里写响应体是手动 `getWriter().write`，不走 HandlerReturnValueHandler，
  因为拦截器阶段 Controller 还没执行


### 2.2 SecurityConfig（Spring Security 配置）

文件：`backend/src/main/java/com/example/minimall/config/SecurityConfig.java`

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig extends WebSecurityConfigurerAdapter {

    @Override
    protected void configure(HttpSecurity http) throws Exception {
        http
            .csrf().disable()  // 前后端分离，token 替代 CSRF
            .cors().and()
            .sessionManagement()

            .and()
            .authorizeRequests()
                .antMatchers("/api/auth/**").permitAll()
                .antMatchers(HttpMethod.GET, "/api/product/list").permitAll()
                .antMatchers(HttpMethod.GET, "/api/product/search").permitAll()
                .antMatchers(HttpMethod.GET, "/api/product/*").permitAll()
                .antMatchers(HttpMethod.GET, "/api/category/**").permitAll()
                .antMatchers("/api/ai/query").permitAll()
                // 微信小程序相关
                .antMatchers("/api/wx/**").permitAll()
                .antMatchers("/uploads/**").permitAll()  // 静态资源
                .anyRequest().authenticated()
            .and()
            .headers().frameOptions().disable();  // 允许 h2-console iframe
    }

    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }

    @Bean
    public CorsConfigurationSource corsConfigurationSource() {
        CorsConfiguration cfg = new CorsConfiguration();
        cfg.setAllowedOriginPatterns(Arrays.asList(
            "http://localhost:5173", "http://localhost:5176",
            "http://localhost:3001", "http://127.0.0.1:*"));
        cfg.setAllowedMethods(Arrays.asList("GET","POST","PUT","DELETE","OPTIONS"));
        cfg.setAllowedHeaders(Arrays.asList("*"));
        cfg.setAllowCredentials(true);
        UrlBasedCorsConfigurationSource src = new UrlBasedCorsConfigurationSource();
        src.registerCorsConfiguration("/**", cfg);
        return src;
    }
}
```

**讲的时候提一句坑：**
> "开发时为了省事我一开始写的是 .anyRequest().permitAll()，
> 测安全的时候才发现这是个洞。`permitAll()` 是 Spring Security 的一个偷懒配置，
> 写代码很快但等于没做权限控制，最终防线还是要靠拦截器层。
> 所以我现在用 Spring Security 做粗粒度的 `authenticated()` 兜底，
> 用 PermissionInterceptor 做细粒度的角色判断。两层防护，缺一不可。"

技术细节：
- `SessionCreationPolicy.STATELESS`：明确不用 HttpSession，全部用 JWT 携带身份，
  这样分布式部署时不用做 session 共享
- `antMatchers("/api/product/*")` 里的 `*` 只匹配单段路径，不含 `/`，
  想匹配多段要用 `**`
- `setAllowedOriginPatterns` 而非 `setAllowedOrigins`，因为后者不支持通配符


### 2.3 下单事务（乐观锁 + 库存扣减）

文件：`backend/src/main/java/com/example/minimall/service/impl/OrderServiceImpl.java`

```java
@Override
@Transactional(isolation = Isolation.REPEATABLE_READ)
public Order createOrder(Long userId, Long productId, 
                          Integer quantity, Long addressId) {
    // 1. 校验地址
    UserAddress addr = addressMapper.selectById(addressId);
    if (addr == null || !addr.getUserId().equals(userId)) {
        throw new BizException("地址无效");
    }

    // 2. 校验商品 + 乐观锁扣库存
    Product p = productMapper.selectById(productId);
    if (p == null || p.getStatus() != 1) {
        throw new BizException("商品已下架");
    }
    if (p.getStock() < quantity) {
        throw new BizException("库存不足");
    }

    // 乐观锁：UPDATE ... WHERE stock = #{oldStock}
    int affected = productMapper.deductStockWithVersion(
        productId, p.getStock(), quantity, p.getVersion());
    if (affected == 0) {
        throw new BizException("库存被其他订单占用，请重试");
    }

    // 3. 创建订单（订单地址做快照）
    Order order = new Order();
    order.setOrderNo(generateOrderNo());
    order.setUserId(userId);
    order.setProductId(productId);
    order.setQuantity(quantity);
    order.setTotalAmount(p.getPrice().multiply(
        new BigDecimal(quantity)));
    order.setStatus(0);  // 待付款
    order.setReceiver(addr.getReceiver());
    order.setPhone(addr.getPhone());
    order.setAddress(addr.getAddress());  // 快照
    order.setCreateTime(LocalDateTime.now());
    orderMapper.insert(order);

    // 4. 清购物车
    cartMapper.deleteByUserIdAndProductId(userId, productId);

    // 5. 异步通知商家（WebSocket 推送）
    websocketService.pushToSeller(p.getSellerId(), 
        "您有新订单：" + order.getOrderNo());

    return order;
}
```

SQL（乐观锁）：
```xml
<update id="deductStockWithVersion">
    UPDATE product 
    SET stock = stock - #{quantity}, version = version + 1
    WHERE id = #{productId}
      AND stock = #{quantityOld}  -- 关键：乐观锁
      AND status = 1
</update>
```

**这块怎么讲：**
> "扣库存是个经典并发坑。如果用 `SET stock = stock - n` 不带条件，
> 10 个线程同时扣，可能都看到 stock=10，扣完变 0，但实际超卖到负数。
> 乐观锁的做法是先查出当前 stock=10 作为 `quantityOld`，UPDATE 时带
> `WHERE stock = 10`，这样只有一个线程 affected=1，其他都 affected=0。
> 失败的就让用户重试。"

> "事务隔离级别用 REPEATABLE READ（MySQL InnoDB 默认），
> 主要防止同事务内多次读 product 表读到不一致的 stock。
> 在这个场景其实用 READ COMMITTED 也行，但默认 REPEATABLE READ 更稳。"

> "还有一个细节：订单表里冗余了 receiver/phone/address 三个字段，
> 是地址的快照。买家后续修改地址，不影响老订单的收货信息显示，
> 这也是电商系统的通用做法。"

> "最后用 WebSocket 异步推送给商家，商家后台能实时看到新订单提示音。"

### 2.4 AI 流式响应（SseEmitter）

```java
@GetMapping(value = "/api/ai/query", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public SseEmitter query(@RequestParam String message,
                         @RequestAttribute(required = false) Long userId) {
    // 1. 创建 emitter，超时 30s
    SseEmitter emitter = new SseEmitter(30_000L);

    // 2. 异步调用 DeepSeek，避免阻塞 Tomcat 线程
    ExecutorService exec = Executors.newSingleThreadExecutor();
    exec.submit(() -> {
        try {
            // 2.1 System Prompt 约束回答范围
            String systemPrompt = "你是「乡农优选」商城 AI 客服，只回答以下"
                + "5 类问题：1) 商品咨询 2) 订单状态 3) 活动优惠 "
                + "4) 售后政策 5) 平台通用规则。其他问题礼貌拒绝。";

            // 2.2 流式调用
            DeepSeekClient.streamChat(systemPrompt, message, new StreamCallback() {
                @Override
                public void onChunk(String chunk) {
                    try {
                        emitter.send(SseEmitter.event()
                            .name("message")
                            .data(chunk));
                    } catch (IOException e) {
                        emitter.completeWithError(e);
                    }
                }
                @Override
                public void onComplete() {
                    emitter.complete();
                }
                @Override
                public void onError(Throwable t) {
                    emitter.completeWithError(t);
                }
            });
        } catch (Exception e) {
            emitter.completeWithError(e);
        }
    });

    return emitter;
}
```

**讲的时候强调三个点：**
> "第一，用 `SseEmitter` 而不是 WebSocket，是因为 AI 回答是单向的（服务端推客户端），
> 不需要客户端发消息，SSE（Server-Sent Events）协议更轻量，
> 基于 HTTP 不用单独起 WS 连接。

> 第二，DeepSeek 返回的是流式的（chunk by chunk），如果用普通 Controller
> 一次性 return，Tomcat 线程会卡到 DeepSeek 全部返回完才释放，并发能力会非常差。
> 改成 `@Async` 提交到线程池后，主线程立即返回 emitter，
> 业务线程边收 chunk 边 send 出去，Tomcat 线程几乎不被占用。

> 第三，30 秒超时是兜底——如果 DeepSeek 那边 hang 住，
> SseEmitter 会自动断开，前端 EventSource 收到 error 事件可以重试。"

```js
  answerText.value += e.data;  // 逐字追加
```

```java
@Controller
public class ChatController {

    @MessageMapping("/chat/{sessionId}")
    @SendTo("/topic/chat/{sessionId}")
    public ChatMessage handleMessage(
        @DestinationVariable Long sessionId,
        ChatMessage message,
        @Header("userId") Long userId) {
```

**讲的时候：**
> "STOMP 是 WebSocket 的子协议，简单讲就是给 WS 加了消息路由的概念。
> 客户端订阅 /topic/chat/123，服务端用 @SendTo 注解就能精准投递。

> 业务代码不用改。"

> "未读数设计：每条消息发送后 incrUnread 对端的 session 行。
> 前端进入会话页时调一个清零接口。"

### 2.6 JWT 工具类

文件：`backend/src/main/java/com/example/minimall/utils/JwtUtil.java`

```java
public class JwtUtil {
    // 从 yml 注入密钥
    private static final String SECRET = 
        SecurityConfig.JWT_SECRET;  // 32 字节以上

    public static String generateToken(Long userId, 
                                        String username, 
                                        Integer userType) {
        Map<String, Object> claims = new HashMap<>();
        claims.put("userId", userId);
        claims.put("username", username);
        claims.put("userType", userType);

        long now = System.currentTimeMillis();
        return Jwts.builder()
            .setClaims(claims)
            .setIssuedAt(new Date(now))
            .setExpiration(new Date(now + 24 * 3600 * 1000))
            .signWith(SignatureAlgorithm.HS256, SECRET)
            .compact();
    }

    public static Claims parseToken(String token) {
        return Jwts.parser()
            .setSigningKey(SECRET)
            .parseClaimsJws(token)
            .getBody();
    }
}
```

> "HS256 是对称加密，签发和校验用同一个密钥。生产环境密钥放配置中心或环境变量，
> 提交到 git 的时候 .gitignore 掉。答辩项目里密钥是写死的
> （application.yml 里的 jwt.secret），演示够用。"

## 三、深度技术解析

### 3.1 为什么是 B/S 架构 + 前后端分离？

**B/S 优点：**
- 不用装客户端，浏览器即用
- 升级只更新服务端，零成本推送
- 跨平台（Win/Mac/手机浏览器都行）

**前后端分离的原因：**
- 后端只暴露 RESTful 接口，前端只是消费者之一
- 同样的接口，Web 商城、小程序、商家后台、管理后台都能用，避免重复开发
- 前端可以独立选型：Web 用 Vue 3 + Vite，小程序用 uni-app / Taro 跨端编译
- 团队分工清晰：前端专攻 UI/UX，后端专攻业务/性能
- 我这个项目实际就是这套：backend/ 一份代码，web-mall/admin-web/seller-web/mini-program 四个前端

**为什么不选 C/S：**
- 每次升级要用户重装客户端
- 不同操作系统要写不同版本（Windows/Mac/Linux 桌面应用）
- 移动端 App 还要上架审核

**为什么不选微服务：**
- 这是毕设项目，4 端共享一份后端，团队就 1 人
- 单体应用开发/部署/调试都简单，DDD 拆服务反而增加复杂度
- 论文 6.6 节"后续工作"里提了可以拆微服务

### 3.2 为什么用 Spring Boot 2.7.9 而不是 3.x？

- 2.7.x 是 Spring Boot 最后一个支持 Java 8 的版本
- 毕设答辩时用 Java 8 更稳，老师机器上一般也是 Java 8
- Spring Boot 3.x 要 Java 17，对开发环境要求高
- 2.7.9 修复了大量 CVE，是 2.7.x 最稳的版本

### 3.3 为什么用 MyBatis-Plus 而不是 JPA/Hibernate？

- 国内生态成熟，中文文档详细
- 自带代码生成器、通用 CRUD、分页插件、乐观锁插件，省事
- 复杂 SQL（联表 5+ 张表）MyBatis 手写可控，JPA 调优难
- 性能上 MyBatis-Plus 比 JPA 略好（少了反射和代理层）
- 我们项目里有大量报表查询，MyBatis 手写 SQL 灵活

### 3.4 为什么用 Redis？

- 缓存：商品详情、分类树、用户 session（虽然用 JWT 但有过渡缓存）
- 计数器：商品浏览数、库存预扣减
- 分布式锁：秒杀时防止超卖（虽然这个项目没用秒杀，但留了接口）
- 排行榜：商家销量榜、热搜商品
- 选 Redis 5.x 是因为公司服务器上稳定跑这个版本，3.x 集群模式有 bug

### 3.5 为什么用 Vue 3 + Vite 而不是 Vue 2 + Webpack？

- Vite 启动 < 1s，HMR 毫秒级，Webpack 要几秒到十几秒
- Vue 3 Composition API 比 Options API 逻辑复用更清晰（`useXxx` 自定义 hook）
- Vue 3 体积小 41%，首屏 FCP 更快
- 答辩演示时启动快、加载快，给老师印象好

### 3.6 为什么用 STOMP 不用原生 WebSocket？

原生 WebSocket 痛点：
- 没有消息路由概念，广播就是 `session.getOpenSessions()` 全发
- 房间/私聊要自己写 sessionId → sessions 映射表
- 心跳、重连、降级（长轮询）全要自己实现
- 业务消息和心跳消息混在同一个 channel

STOMP 优势：
- 帧协议自带 message/headers/body 结构，业务和心跳分离
- 订阅模型 `/topic/xxx` 自动路由
- 注解开发，`@MessageMapping` / `@SendTo` 一行代码完成
- SockJS 降级内置

### 3.7 为什么用 JWT 不用 Session？

Session 痛点：
- 服务器要存 session Map，分布式部署要 session 共享（Redis 或 sticky session）
- 跨域麻烦，cookie 跨不了 domain
- CSRF 攻击面，token 又要额外配

JWT 优势：
- 无状态，服务器不用存，解析后直接信任
- 自包含（userId、role 全在 token 里）
- 天然支持跨域（放 Authorization header）
- 移动端友好（App 也能用同一个 token）

JWT 注意事项：
- 密钥不能泄露，签发后无法主动失效（除非用黑名单）
- payload 不要放敏感信息（明文密码、手机号），Base64 可解码
- HTTPS 必上，不然明文传输等于裸奔

### 3.8 为什么订单地址做快照？

电商系统里订单表冗余 `receiver / phone / address` 三个字段，
不直接外键关联 `user_address` 表，理由：

1. **历史数据稳定**：买家下单后改了地址，不应影响老订单的显示
2. **查询效率**：订单列表只查一张表，不用 join address
3. **地址删除兼容**：买家可以删除某个地址，但订单依然能展示历史收货信息
4. **审计需要**：售后介入时不能因为买家改地址就看不到原收货地址

代价是数据冗余和修改地址不会回写——但这两个代价对电商系统完全可接受。

### 3.9 为什么用乐观锁不用悲观锁（`SELECT ... FOR UPDATE`）？

| 维度 | 乐观锁 | 悲观锁（FOR UPDATE） |
|---|---|---|
| 实现 | UPDATE WHERE version=? | SELECT ... FOR UPDATE |
| 锁粒度 | 无锁，CAS | 行级 X 锁 |
| 适用 | 读多写少、冲突少 | 写多、冲突频繁 |
| 性能 | 高（无锁等待） | 低（事务等待） |
| 失败处理 | 业务层重试 | 锁等待超时 |

下单扣库存场景，99% 的请求不会冲突，乐观锁最优。
悲观锁会让 50 个用户同时下单时，全部串行化等锁，QPS 暴跌。

### 3.10 AI 客服的局限性

- 没用 RAG：知识全靠 LLM 预训练，平台特定政策（"本平台满 99 包邮"）答不准
- 没有多轮上下文管理：每次请求都带全量 history，token 消耗大
- System Prompt 是硬编码的，运营改文案要重新部署
- 这些都在论文 6.6 节"后续工作"里列了

**Q20: 怎么做的接口幂等？**
A: 订单创建时生成唯一订单号 `ORD + yyyyMMddHHmmssSSS + 6位随机数`，
数据库 UNIQUE 约束。重复提交时订单号冲突会抛异常回滚。
支付回调用同一订单号去重。

**Q21: 怎么处理大文件上传？**
A: 当前用的是单文件 multipart 上传，限制 5MB。
如果要支持大文件，需要：
1. 前端分片（每片 5MB）
2. 后端接收分片 + 合并 + MD5 校验
3. 断点续传（前端记已上传分片 index）

**Q22: 实时聊天的消息可靠吗？**
A: STOMP 消息先落库再推送。断线重连时拉取未读消息补齐。
但当前实现没做 ACK 确认，理论上有消息丢失风险，后续可加 STOMP receipt 头。

**Q23: WebSocket 怎么鉴权？**
A: 客户端在 STOMP CONNECT 帧里带 `Authorization: Bearer xxx`，
WebSocketConfig 里配 ChannelInterceptor 拦截 CONNECT 帧，
验证 token 合法性后将 userId 放到 Principal 里。
消息级别不再二次鉴权。

### 安全类

**Q41: SQL 注入怎么防的？**

### 数据库深入类

**Q51: 你的 37 张表，表之间的关系是怎样的？**
A: 核心关系链有六条：
1. user → cart → product → orders → order_item（用户→购物车→商品→订单→订单明细）
2. user → chat_session → chat_message（用户→聊天会话→聊天消息）
3. product → product_spec / product_image / product_review（商品及其规格/图片/评价）
4. user → after_sale_service → service_record（售后申请→处理记录）
5. category 自关联（parent_id 指向自己，形成树形结构）
6. product_tag_relation 关联 product 和 product_tag（多对多中间表）

**Q52: 为什么用自关联做分类树而不是嵌套集或路径枚举？**
A: 三种方案的取舍：
- **邻接表（自关联 parent_id）**：最简单的实现，增删改查都好理解，缺点是查整棵树要递归。我的分类只有 3 层、最多几十个节点，递归查询完全够用
- **嵌套集（left/right 值）**：查子树 O(1) 但是插新节点要更新所有 left/right 值，太复杂
- **路径枚举（"1/2/5"）**：修改分类时需要批量更新路径字符串
结论：毕设项目节点少、修改少，自关联最简单最稳

**Q53: 你的索引是怎么设计的？哪些字段加了索引？**
A: 遵循"等值查询加索引、范围查询放在后、联表字段必加索引"原则：
- `user` 表：phone 唯一索引（手机号登录）、username 唯一索引、openid 索引（微信登录）、userType 索引（角色筛选）
- `product` 表：name FULLTEXT 索引（商品搜索）、category_id 索引（按分类筛选）、seller_id 索引（商家查自己商品）
- `orders` 表：user_id 索引（我的订单）、order_no 唯一索引（订单号查重）、status 索引（按状态筛选）、create_time 索引（按时间排序）
- `chat_session` 表：user_id 和 seller_id 复合索引（查双方的会话列表）
- `chat_message` 表：session_id 索引（查会话消息）、created_at 索引（消息排序）
- 联表字段：product_id、order_id、session_id 等外键全部加了索引

**Q54: 你的数据库设计符合第几范式？为什么要反范式设计？**
A: 大部分表符合第三范式（3NF），但有三处故意的反范式设计：
1. **订单表冗余了收货地址**（receiver/phone/address）：原因在 Q16 讲过——历史数据稳定、查询效率、地址删除兼容
2. **chat_session 冗余了 lastMessage 和 unreadCount**：每次显示会话列表都去 chat_message 表 count(*) 太慢，冗余到会话表里，发消息时同步更新
3. **admin_intervention 冗余了 buyerName/sellerName/orderNo/productName**：纠纷记录需要"当时的快照"，不能因为用户后来改名就看不到原记录
这些都是"用空间换时间"和"用冗余换稳定"的经典场景

**Q55: 数据库表的字符集和排序规则怎么选的？**
A: 全部使用 `utf8mb4` + `utf8mb4_unicode_ci`。
为什么要 `utf8mb4` 而不是 `utf8`？MySQL 的 `utf8` 其实是阉割版，最多 3 字节，存不了 emoji（😂 是 4 字节）和部分生僻字。`utf8mb4` 是完整的 UTF-8，最多 4 字节。
生鲜商城里用户评价可能发 emoji（"苹果真好吃🍎"），必须用 `utf8mb4` 才能正常存储。

### Redis 深入类

**Q56: 缓存穿透、缓存击穿、缓存雪崩分别是什么意思？你怎么处理的？**
A: 这是面试高频问题，答辩老师也可能问：
- **缓存穿透**：恶意请求不存在的商品 ID（比如 id=-1），Redis 里没有，每次穿透到 MySQL。用**布隆过滤器**或把"不存在"也缓存（value 设为 null，TTL 短一些）
- **缓存击穿**：热点商品缓存刚好过期，瞬间大量请求同时打到 MySQL。用**互斥锁**——第一个请求去查库并回写缓存，其他请求等待这把锁释放后读缓存
- **缓存雪崩**：大量缓存在同一时间集体过期，MySQL 瞬间压力过大。用**随机化 TTL**——不要把 1000 个商品的缓存全设 30s，加随机偏移（30 + random(0, 10)），让过期时间分散开
在本项目中：热门商品详情用了互斥锁防击穿；TTL 设置了随机偏移防雪崩；暂时没用布隆过滤器（后续可加）

**Q57: Redis 在你的项目里是单机还是集群？为什么？**
A: 单机。毕设项目用户量小，单机 Redis 已经能扛 10 万 QPS，加集群反而增加复杂度。论文 6.6 节写了未来可升级到 Redis Cluster 或哨兵模式。

**Q58: Redis 和 MySQL 的数据一致性怎么保证？**
A: 用了"先更新数据库，再删除缓存"的策略（Cache Aside Pattern）：
1. 更新商品时：`UPDATE product SET ... → DEL product:{id}`（先写库，后删缓存）
2. 查商品时：先查 Redis，命中返回；未命中查 MySQL，回写 Redis
为什么不是"先删缓存再更新数据库"？因为并发下可能出现——A 删缓存→B 读缓存 miss 查库拿到旧值写回→A 更新库→缓存里是旧值，数据不一致。
先写库再删缓存能最大化减少不一致窗口，极端情况下的脏数据也会随着 TTL 过期被自动修。

### 前端与小程序类

**Q59: 你的四个前端工程是多页面应用（MPA）还是单页面应用（SPA）？为什么？**
A: SPA（Vue Router 管理所有页面，一个 HTML 承载整个应用）。选 SPA 的理由：
- 页面切换无白屏（JS 动态替换内容，不用重新加载整个 HTML）
- 状态可以在页面间共享（Pinia Store 存用户信息，所有页面都能访问）
- 配合后端 RESTful API，天然适合前后端分离
代价是首屏加载稍慢（要下载 JS 体积），但 Vite Tree Shaking 已经优化过，实际可接受。

**Q60: 你的微信小程序用的是什么框架？怎么和后端通信的？**
A: 微信原生框架，用 `wx.request()` 发 HTTPS 请求到后端 API。和后端通信和 Web 端走同一套接口（`/api/auth/login`、`/api/product/list` 等），后端只写一份接口代码，Web 和小程序共享。区别是小程序端不支持 cookie，JWT token 存 `wx.setStorageSync('token', xxx)`。

**Q61: 四个前端工程之间有没有共享代码？怎么共享的？**
A: 有。`shared-ui/` 目录放了公共 UI 组件（比如商品卡片、评价星星、价格格式化），用 npm workspace 管理，多个前端工程通过 `import` 引用。这样改一个组件四个工程同步更新，不用改四次。

**Q62: 前端的响应式适配做了吗？手机端和桌面端怎么处理的？**
A: Web 端用 CSS 媒体查询 + Element Plus 的响应式栅格做了基础适配（手机端单列、平板两列、桌面三列显示商品）。但实际上桌面端和小程序是两套独立的前端工程——桌面端用 Vue 3 浏览器运行，手机场景走微信小程序（原生体验更好而非手机浏览器）。

**Q68: 你的代码中有没有用设计模式？用了哪些？**
A: 用了四种：
1. **单例模式**：Spring 的 Bean 默认就是单例（`@Service`、`@Controller` 全局只有一个实例）
2. **工厂模式**：`Result.success()` / `Result.error()` 是静态工厂方法，比 `new Result()` 然后用 setter 设值更语义化
3. **策略模式**：`PasswordValidator.Rule` 枚举，每个枚举值代表一种密码校验规则（长度、大小写、数字、特殊字符），用 `Arrays.stream(Rule.values())` 遍历执行
4. **模板方法模式**：`BaseController` 提供 `success()` / `error()` 模板方法，子类继承复用
5. **拦截器模式（责任链）**：ApiLogging → Performance → RateLimit → Permission 四个拦截器按顺序执行

### HTTP 与 RESTful 类

**Q69: 你的接口设计符合 RESTful 规范吗？举几个例子。**
A: 基本符合，遵循了"用 HTTP 方法表达操作、URL 表达资源"的原则：
- `GET /api/product/{id}` → 查商品（幂等、无副作用）
- `POST /api/order/create` → 创建订单（非幂等、有副作用）
- `PUT /api/product/{id}` → 更新商品（幂等）
- `DELETE /api/product/{id}` → 删除商品（幂等）
状态码也规范：200 成功、400 参数错误、401 未登录、403 权限不足、404 不存在、429 频率限制、500 服务器错误。不过有些接口为了兼容小程序用了 `POST /api/order/cancel/{id}` 而不是 `PUT /api/order/{id}/status`（把动作写在 URL 里），不够纯 RESTful，但兼容性优先。

**Q70: 你的接口有做版本管理吗？**
A: 当前没有（URL 里没有 `/v1/` 前缀）。因为项目还在毕业设计阶段，不需要向后兼容。如果以后要上线，会在路径里加 `/api/v1/` 和 `/api/v2/`，通过 Nginx 路由分发不同版本的请求到不同后端服务。

### 项目心得与收获类

**Q71: 做完这个项目你最大的收获是什么？**
A: 三个：
1. **动手验证理论**：教科书上说乐观锁、JWT、缓存这些概念，真正写代码实现才知道细节多重要（比如乐观锁的 `WHERE stock = oldStock` 漏写就变成悲观锁，JWT 密钥写死在本地的安全问题）
2. **测试不是"测完了"而是"测出问题了再改"**：最初 4 个安全用例全挂，不是因为代码不会写，而是因为设计模式选了"默认放行"。改成了"默认拒绝"后才全过。测试的价值在于推动设计改进
3. **前后端分离不是说"分开了就行"，接口文档和沟通更重要**：遇到过前端调 `productId`，后端返回的是 `product_id`，查了半天才发现字段名约定不一致。后来统一用 camelCase 并写了 Swagger 接口文档

**Q72: 如果重新开始这个项目，你会做什么不同的选择？**
A: 三点：
1. 先写测试再写代码（TDD）：如果从下单接口的测试用例开始写，乐观锁的设计一开始就不会漏
2. 用 Swagger 生成前端 API 调用代码：手写 Axios 请求容易拼错 URL，自动生成省时且类型安全
3. 小程序用 uni-app 或 Taro 跨端框架：现在小程序和 Web 是两套独立代码，用跨端框架可以只写一套就生成 H5 + 小程序两端代码

**Q73: 你从这个项目中学到了哪些"工程化"的东西？**
A: 版本管理（Git 分支策略）、自动化测试（mvn test 集成到开发流程）、日志监控（traceId 全链路追踪）、Docker 环境隔离、接口文档规范（Swagger）、代码规范（Lombok 消除样板代码、常量化魔法值）。这些都不是课程教的，是在实际写代码过程中发现"没有这些工具项目会越来越难维护"之后自己学的。

**Q74: 你觉得这个项目还有哪些不足？**
A: 诚实地说四点：
1. 支付是模拟的，没对接真实微信支付/支付宝
2. AI 客服没有 RAG（检索增强生成），平台自己的政策答不准
3. 没有做 CDN，商品图片直接走服务器，用户多了带宽和存储都是瓶颈
4. WebSocket 是单机内存模式，没有用 Redis Pub/Sub 做跨节点广播，多台服务器部署时消息会丢
但这些都在论文 6.6 节"后续工作"里写了改进方案。

**Q75: 老师可能会问"你这个项目上线了吗？有人用吗？"**
A: 目前是毕设演示阶段，没有正式上线。但已经在本地跑通了四个前端 + 一个后端的全流程，数据库里有真实的商品和分类数据，测试脚本验证了 50 并发的性能。如果需要上线，需要：申请云服务器、配置域名和 HTTPS、对接支付和物流 API、做安全加固。这些都是工程上的平移，核心业务逻辑已经验证过了。

### 8.3 下单（最关键的事务流程）

```
┌──────────────────────────────────────────────────────────────┐
│                       下单事务流程                            │
└──────────────────────────────────────────────────────────────┘

买家在购物车点"去结算"
  ──> POST /api/order/create
       Body: {items: [{productId, quantity}], addressId, remark}
  ──> OrderController.create
  ──> @Transactional ──> OrderService.createOrder()
       │
       │  1. 校验：所有商品 status=1（已上架）、库存 >= 购买量
       │  2. 加锁：SELECT * FROM product WHERE id IN (...) FOR UPDATE
       │     （悲观锁，防超卖）
       │  3. 扣库存：UPDATE product SET stock = stock - #{qty}
       │  4. 计算金额：遍历 items 累加 price * qty
       │  5. 生成 order 记录，status=0（待支付）
       │  6. 生成 order_item 记录（一条订单多个商品）
       │
       │  任何一个步骤抛异常 ──> @Transactional 自动回滚
       │  ──> 库存恢复 + 订单不生成
       │
  ──> 返回 {orderId, totalAmount, expireTime: now+15min}
  ──> 跳到支付页（演示版直接调 mock 支付，2 秒后回调）
  ──> POST /api/order/pay {orderId} ──> status=1（已支付）
  ──> 增加商品销量 saleCount += quantity
```

**关键代码**：
- [OrderServiceImpl.java](file:///e:/迅雷下载/mall_system_extended/backend/src/main/java/com/example/minimall/service/impl/OrderServiceImpl.java) — `createOrder` 方法
- `SELECT ... FOR UPDATE` 在 [OrderMapper.xml](file:///e:/迅雷下载/mall_system_extended/backend/src/main/resources/mapper/OrderMapper.xml) 中
- `@Transactional` 注解保证原子性

**答辩话术**：
> "下单用 Spring 声明式事务 @Transactional 包裹，6 个步骤全部成功才提交，中间任何抛异常都回滚。防超卖靠 SELECT FOR UPDATE 悲观锁，多个买家同时下单时排队处理，单机演示 50 并发无超卖。"

### 8.4 实时聊天（WebSocket + STOMP + AI）

## 九、补充高频问题（Q76 ~ Q113）

> 以下问题基于项目代码实际分析得出，覆盖了现有 75 问之外的高频追问点。

### 9.1 关于近期发现的代码问题

**Q76: 刚才演示后台仪表盘时页面白屏（403），能解释原因吗？**
A: 这是权限拦截器的白名单遗漏。SystemController 的路径 `/api/system/dashboard/stats` 不在卖家角色白名单中。原因是 `isSellerAllowed()` 只配了 `/api/dashboard/seller`，而 SystemController 的 RequestMapping 前缀是 `/api/system/`，不匹配。加一行 `startsWith("/api/system/dashboard/")` 解决。本质是默认拒绝模式下白名单未覆盖新接口的典型场景。

**Q77: 如果没人指出这个 Bug，你自己排查的思路是什么？**
A: 三步：1）Network 面板看返回的是 403 而非 401，排除 token 问题；2）看后端控制台有没有 `PermissionInterceptor` 打印的 "卖家无权访问该接口" 日志；3）定位到 `isSellerAllowed()` 的方法，对照 SystemController 的 `@RequestMapping` 逐条比对白名单。整个排查 2-3 分钟能完成。

### 9.2 关于多端多端口（答辩现场高频追问）

**Q78: 你为什么把 admin-web 放 3001、web-mall 放 5176、seller-web 放 5173，而不是 3001/3002/3003？**
A: Vite 默认端口是 5173，seller-web 用了默认值 5173，web-mall 用了 5176，admin-web 为了跟其他两个区分开设了 3001。没有统一规划，是开发过程中逐步加的。真上线时 nginx 反向代理后端口对外不可见，内部用什么端口不影响用户体验。

**Q79: 如果三个前端跑在同一个端口，会有什么问题？**
A: 跑不了。同一台机器上不能有两个进程监听同一个 TCP 端口，后启动的会报 `Port already in use`。三个前端是三个独立 Vite Dev Server 进程，必须不同端口。这是操作系统层面的硬限制，不是设计选择。

### 9.3 关于代码与论文的对应

**Q80: 你论文里 SystemController 的仪表盘功能，属于哪个业务模块？为什么模块划分图里没有它？**
A: SystemController 不属于任何一个业务模块（商品/订单/聊天/售后），而是系统运维层面的辅助功能。论文的模块划分是按"购物与订单管理""售后服务""AI 智能客服"等业务线来的，仪表盘在论文 4.x 章的管理后台小节中描述，没有单独列为一个模块。答辩时的模块划分图里应该补上"系统管理"这个横切模块。

**Q81: 论文里 21 条测试用例和项目里 145 个 JUnit 测试是什么关系？**
A: 论文只列了具有代表性的 21 条（U-01 ~ U-21、P-01 ~ P-05、S-01 ~ S-04、E2E-01/02、PF-01 ~ PF-04），每条都有详细的测试步骤和预期结果。项目里 145 个 JUnit 用例是完整单元测试 + Controller 测试 + 集成测试的总和。两者是"精华版 vs 完整版"的关系，21 条是可以在答辩现场 5 分钟内演示完毕的精简版。

### 9.4 关于 AI 辅助开发（2026 年答辩必问）

**Q82: 你在这个项目里用了哪些 AI 工具？占比多少？你怎么审核 AI 生成的代码？**
A: 用 AI 辅助了三个场景：1）MyBatis Mapper XML 模板；2）Vue 组件骨架代码；3）JUnit 边界值测试数据。占比约 20-30%。所有 AI 生成的代码走四步审核：人工审阅逻辑正确性 → 编译通过 → 跑测试验证 → Git diff 逐行对比。核心业务逻辑（PermissionInterceptor 默认拒绝模式、下单事务乐观锁、SseEmitter 流式推流、DeepSeek System Prompt 工程、STOMP 消息路由）全部手写。AI 写不出跟项目上下文耦合紧密的代码。

**Q83: 老师怎么判断你的代码不是 AI 全自动生成的？**
A: 三个证据：1）Git 提交记录的渐进式迭代（从默认放行→默认拒绝经历多次提交）；2）踩坑记录（WebSocket 跨域调了一天发现是 `setAllowedOriginPatterns` 和 `setAllowedOrigins` 的区别——AI 不会踩这种坑）；3）可以现场讲解任意一段核心代码的设计思路。AI 生成的代码通常"看起来完整但细节经不起推敲"。

### 9.5 关于系统真实可用性

**Q84: 你的支付和物流都是模拟的，售后退款能真的打钱吗？**
A: 不能。支付和物流是外部依赖，打钱需要对接支付宝/微信支付的商户接口。毕设核心价值是验证业务逻辑正确性——事务原子性、状态机流转、权限控制闭环。真上线时对接支付 SDK：注册商户号 → 调统一下单接口 → 异步回调验签，核心业务代码不需要改。

**Q85: 如果让你推荐一家真的生鲜小店用你的系统，你敢吗？**
A: 不敢，至少目前不敢。缺三个关键环节：1）HTTPS + 域名 + CDN；2）专业安全渗透测试（SQL 注入/XSS/CSRF 只做了基础防护）；3）运维保障（单机部署无备份无容灾）。但作为 MVP 原型是合格的，再经 2-3 个月工程化加固可考虑小范围试用。

**Q86: 你的系统有"助农"元素吗？论文标题里写的是"乡农优选"，实际系统里助农体现在哪里？**
A: 体现在三处：1）activity 表支持"大宗采购""农场参观""实地观光"三种助农活动类型，不只是卖商品；2）首页推荐机制（is_recommended + recommend_order）优先展示助农产品；3）论文里把"生鲜农产品"这个领域和"助农电商"的政策导向结合，但说实话，业务层面没有做深度助农功能（比如产地溯源、农户直连、帮扶数据看板），这是一个可以追问的点。答辩时坦承就好，后续可以加。

### 9.6 关于数据库设计细节

**Q87: 你 37 张表有没有用外键约束？为什么？**
A: 没有用数据库层面的物理外键。理由三：1）分库分表时外键无法跨库；2）外键级联删除有风险（误删一条 category 可能级联删掉所有关联 product）；3）一致性交给应用层（Service 事务 + 乐观锁）。代价是可能产生脏数据，但通过代码 Review 和集成测试弥补。阿里 Java 开发手册明确"不得使用外键与级联更新"。

**Q88: category 表的 is_deleted 字段选择性极低（只有 0/1 两个值），单独建索引有意义吗？**
A: 单独建索引几乎没有意义——MySQL 优化器可能直接全表扫描。我的做法是把 is_deleted 放在组合索引最后：`INDEX (parent_id, sort, is_deleted)`，先用高选择性字段缩小范围。MySQL 对 low-cardinality 字段放复合索引末尾是标准做法。

**Q89: activity 表的 current_participants 冗余字段你说防高频 COUNT，但报名和退订并发时计数器准不准？**
A: 准的。UPDATE 语句带行级排他锁：`UPDATE activity SET current_participants = current_participants ± 1 WHERE id = #{id}`。两个并发操作会串行执行，不会算错。极端情况是事务回滚时无法撤销计数——我的防护是在 `@Transactional` 内先做业务校验（库存/重复报名），确认都通过后再加计数和插记录，校验失败在改计数之前就抛异常回滚了。

### 9.7 关于微服务与部署

**Q90: 你说后续可以拆微服务，具体拆哪几个、怎么通信？**
A: 拆 5 个服务：用户服务（注册/登录/JWT）、商品服务（CRUD/库存/分类）、交易服务（订单/支付/售后/纠纷）、聊天服务（WebSocket/STOMP/AI）、运营服务（活动/优惠券/仪表盘）。同步通信：Spring Cloud Gateway + OpenFeign（比如订单服务查用户信息调用户服务）；异步通信：RabbitMQ（支付回调通知、库存同步、聊天消息落库）。注册中心/配置中心：Nacos。

**Q91: 你的 docker-compose.yml 里 backend 端口是 8080，实际开发跑 8081，为什么？**
A: docker-compose.yml 是模板用 8080，本地开发时通过 `SERVER_PORT=8081` 环境变量覆盖了，因为 8080 可能被其他程序占用。配置管理不规范——两个地方不同步。生产部署时应统一为 8080，前端 Nginx 反向代理到 8080。

**Q92: Redis 在你系统里是单点吗？如果它挂了会怎样？**
A: 降级但不崩溃。1）缓存 miss → 回源 MySQL，慢但功能正常；2）限流拦截器 catch 异常后 `return true` 放行（`RateLimitInterceptor` 代码里写了降级逻辑）；3）聊天在线状态丢失 → 所有人显示"离线"，但消息仍落库并正常推送。设计原则：Redis 是辅助优化，不是单点故障源。

### 9.8 关于性能压测

**Q93: 你的压测只测了 4 个读接口，为什么没测下单接口的并发？**
A: 下单涉及事务和锁，脚本准备复杂（要先造用户/地址/购物车数据）。Python ThreadPoolExecutor 的线程开销大，测出来数据参考价值有限。论文如实只列了读接口的性能数据。如果需要，后续用 JMeter 或 wrk2 做混合读写场景压测。

**Q93 追问：如果下单接口要做压测，你怎么设计？**
A: 100 并发 / 30 秒持续 / 每个线程独立 userId + productId（避免锁冲突）。三类指标：成功率、P50/P95/P99 延迟、库存最终一致性（并发结束后 product.stock = 初始值 - 成功下单数）。

**Q94: 你论文里商品列表 95ms，压测脚本跑出来是 672ms，给老师演示时怎么解释？**
A: 95ms 是单次请求、连接池预热后的稳定值。672ms 是 50 并发 + 冷启动 + HikariCP 未就绪。先做 5 次预热跑再压，实际能到 300ms 左右。如果老师追问"为什么差这么多"：连接池默认 10 个连接，50 并发时 40 个在排队等连接，增加最大连接数可以改善。

### 9.9 关于安全细节

**Q95: BCrypt 的 cost factor 你设了多少？为什么是这个值？**
A: 10（BCrypt 默认值，2^10 = 1024 次迭代）。单次耗时约 100-200ms，登录场景可接受。cost=12 安全一点但耗时翻倍，用户登录感受明显卡顿。如果未来升级可以切到 Argon2id，同耗时下安全性更高。

**Q96: JWT Secret 你打算生产环境放哪里？**
A: 环境变量 `${JWT_SECRET}`，通过 Docker Compose environment 或 K8s Secret 注入，绝对不提交到 Git。目前 application.yml 里写死的是演示环境。

**Q97: XssFilter 能绕过吗？**
A: 能。双编码、Unicode 编码等高级 XSS 手法可能绕过去。我的防护是两道：XssFilter（入参过滤）+ Vue 3 的 `{{ }}` 默认插值转义（输出过滤）。完全防住需要 CSP 响应头 + 定期渗透测试。论文里承认这不是全覆盖。

**Q98: 你的系统对 CSRF 有防护吗？**
A: JWT 存在 localStorage 而非 cookie，天然的 CSRF 风险较低（攻击者无法让浏览器自动携带 token）。但如果 XSS 被突破，攻击者能读到 localStorage 中的 JWT 发起伪造请求。所以防 CSRF 的前提是防 XSS。

### 9.10 关于微信小程序

**Q99: 你的小程序能跑吗？为什么不现场演示？**
A: 代码写好了，但要前条件：微信开发者工具注册 AppID、配置服务器域名白名单、HTTPS 证书。答辩现场网络和时间不一定允许。优先演示"Web 端和小程序共用同一套后端接口"的架构设计。

**Q100: 小程序和 Web 端共用后端接口，有什么兼容性坑？**
A: 最大的坑是认证机制。Web 端用 Axios interceptor + localStorage，小程序用封装的 `wx.request` + `wx.setStorageSync`。两套认证逻辑独立维护，如果后端改了 token 传参方式，要改两边，容易遗漏。这是多端架构的固有代价。

### 9.11 关于代码架构

**Q101: 你的 Backend 只有 com.example.minimall 一个包，27 个 Controller 全塞在里面不臃肿吗？**
A: 确实臃肿，这是单模块的代价。重构方案：按业务域拆 5 个子包——`user`（用户+认证）、`product`（商品+分类）、`trade`（订单+支付+售后+纠纷）、`chat`（聊天+介入）、`operation`（活动+优惠券+仪表盘）。但包结构只是代码组织问题，不影响功能正确性。

**Q102: shared-ui 里 5 个组件四个前端都用了哪些？为什么不发 npm 包？**
A: 用了按钮、卡片、输入框、弹窗、标签，通过 Vite alias 直接引用，没发 npm 包。四个前端都本地开发，发包没意义，改组件后所有前端实时看到效果就行。不过严格说 shared-ui 更多是"设计规范 + CSS 变量"而非"组件库"，因为业务组件（商品卡片、卖家标签）本身就依赖项目接口类型。

### 9.12 关于业务逻辑细节

**Q103: 你的售后状态机有没有"死锁"状态——即某个状态下无法前进、也无法回退？**
A: 有隐患。如果商家在拒绝退款时，买家刚好提交了退货物流（并发操作），可能出现状态不一致。目前靠前端按钮状态控制（操作后按钮变灰），没有用乐观锁或分布式锁保护状态变更。这是一个应该修的 bug。

**Q104: 15 分钟未支付的订单你怎么处理？**
A: 当前用 `@Scheduled(fixedRate = 60000)` 每分钟扫一次 `status=0 AND create_time < now() - 15min`，改状态为已取消 + 回补库存。隐患：`@Scheduled` 默认单线程，扫表耗时超 1 分钟会丢调度。更好做法：Redis KeyExpiration + 消息队列延迟消息。

**Q105: 聊天消息的已读/未读计数跟你实际用两个窗口时会怎样？**
A: 有 bug。当前按会话维度记 `unread_count`，如果接收者同时开两个窗口进同一个会话，计数会被错误清零。正确做法是按用户维度记未读数，前端区分"全局未读"和"会话未读"。

### 9.13 按评委可能的研究方向准备

**Q106:（数据库方向评委）你的订单表没有拆成主表和明细表，是一个 Bug 还是设计上就这样？**
A: 当前下单接口（`OrderController.create`）是一次买一个商品，`Body: {productId, quantity}`，所以 productId/quantity 直接冗余在 orders 主表里，order_item 表实际没用到。要支持"购物车多选批量下单"时，必须把 order_item 表用起来。论文 E-R 图里画了两张表，代码实现简化了——答辩时诚实说就好。

**Q107:（前端方向评委）你的四个前端做了无障碍访问吗？**
A: 完全没做。没有 ARIA 标签、没有键盘导航、没有屏幕阅读器兼容。电商 2C 产品无障碍是必须的，但毕设优先级放到了功能完整性上，论文里没有写这一块。

**Q108:（分布式方向评委）你的 IM 聊天在多台服务器部署时消息为什么会丢？解决思路是什么？**
A: STOMP broker 是 `SimpleBroker`（内存模式），用户 A 连机器 1、用户 B 连机器 2 时，机器 1 的 broker 找不到 B 的连接。解决：用 Redis Pub/Sub 做跨节点消息广播——每台机器订阅 Redis 频道，发消息时除了本地推送也往 Redis 发一份，其他机器收到后查本地是否有目标用户连接再推送。

**Q109:（什么都问的评委）你这个项目写了多少行代码？**
A: 后端 Java 约 12000 行（含 Mapper XML + SQL），前端 Vue/TS/CSS 约 15000 行，合计 25000-30000 行。测试代码约 2000 行。更准确的说法：37 个 Entity、27 个 Controller、28 个 Service、37 个 Mapper、前端约 80 个页面/组件。

**Q110: 论文里最弱的一部分是什么？如果老师重点挑刺，你怎么回应？**
A: 最弱的是性能测试——只测了 4 个只读接口，没测写操作。我会坦承并补充：后续需要混合场景压测、热点代码分析、长时间浸泡测试。论文 6.6 节已列了改进方向。

**Q111: 如果评委说"你这个跟淘宝比就是个玩具"，你怎么回应？**
A: 确实是教学项目，定位是毕设不是商业产品。但教学项目要展示的是完整的工程能力链（需求→设计→实现→测试→部署），不是跟 BAT 比 QPS。我们有 21 条真实可复现测试、四端联动、AI 真实集成、安全漏洞发现与修复过程，已经能说明独立完成中等规模项目的能力。

**Q112: 做完这个项目最大的收获是什么？**
A: 三个：1）动手验证理论——乐观锁、JWT、缓存这些概念真正写代码才知细节多重要；2）测试的价值不是"测完了"而是"测出问题再改"——从默认放行到默认拒绝是测试驱动的设计改进；3）前后端分离不是说"分开了就行"，接口文档和命名约定（camelCase vs snake_case）才是真正的沟通成本。

**Q113: 如果重新来一次，你会做什么不同的选择？**
A: 三点：1）先写测试再写代码（TDD），乐观锁设计一开始就不会漏；2）用 Swagger/OpenAPI 生成前端 API 调用代码，避免手写 Axios URL 拼错；3）小程序用 uni-app 或 Taro 跨端框架，一套代码跑 H5 + 小程序两端。
